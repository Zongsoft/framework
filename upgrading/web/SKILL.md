# Zongsoft Upgrading Web

## Start Here

Read these local files before changing behavior:

- `README.md` for the package manager contract.
- `docs/index.md`, `docs/api.md`, `docs/models.md`, `docs/requirements.md`, and `docs/planning.md` for design context.
- `Controllers/UpgraderController.cs` for discovery and evaluator endpoints.
- `Controllers/ReleaseController.cs` for manifest import and package upload routes.
- `Services/ReleaseService.cs` for import, upload, validation, application and edition maintenance, and subservices.
- `Upgrader.cs` for release filtering and conversion to client `Release` objects.
- `Evaluator*.cs` and `Module.cs` for evaluator registration and module state.
- `Settings.cs` and `Zongsoft.Upgrading.Web.option` for storage and database settings.
- `../database/zongsoft.upgrading-*.sql` when changing schema assumptions.

## Core Contract

Preserve the Web package manager responsibilities:

1. Store application, edition, release, release property, release executor, instance, and publishing metadata.
2. Import `.manifest` files generated by `dotnet-upgrade pack`.
3. Upload package content for existing releases into configured Zongsoft file-system storage.
4. Recalculate stored package size and checksum during upload.
5. Expose management services through Zongsoft service controller conventions.
6. Serve the client discovery endpoint used by `Zongsoft.Upgrading.Upgrader`.
7. Apply optional evaluator checks before returning releases.

## Discovery Rules

Keep the upgrader endpoint compatible with:

```http
GET /Upgrading/Upgrader/{name}/{edition?}?Platform=Windows&Architecture=X64&CurrentlyVersion=1.0.0
```

Returned releases must be:

- visible;
- published;
- not deprecated;
- matching application name, platform, and architecture;
- matching the requested edition, or edition-less when the edition is empty or `_`;
- newer than `CurrentlyVersion` when provided;
- not newer than `UpgradingVersion` when provided;
- associated with an existing package path and positive package size.

Treat query parameters and request headers as evaluator input. Evaluators may use hardware fingerprint fields sent by the client upgrader.

## Management Rules

When changing manifest import, preserve property and executor creation from the manifest and keep application or edition records synchronized.

When changing package upload, preserve storage path generation from release identity:

```text
{storage}/{lowercase-release-name}/{release-name}[-edition]@{version}_{runtime}
```

When changing publishing state, preserve compatibility with the tool's Web publish flow:

1. POST manifest import with `format=manifest`.
2. Upload package content to the imported release.
3. PATCH or update `Published` to `true`.

## Evaluators

Register custom evaluators in `Module.Current.Evaluators`. Keep evaluator names case-insensitive and return only releases whose configured evaluator succeeds.

Expose evaluator metadata through:

```http
GET /Upgrading/Upgrader/Evaluators
```

## Validation

Prefer focused validation first:

```shell
dotnet build Zongsoft.Upgrading.Web.slnx
```

For API changes, update matching files under `docs/http/` and verify at least the discovery endpoint, manifest import, package upload, and publish-state update flow against a local development instance when available.
