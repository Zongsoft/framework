## Overview

[**Z**ongsoft.**U**pgrading](https://github.com/Zongsoft/framework/tree/main/upgrading) is the automatic application upgrade library for the [_**Z**ongsoft_](https://github.com/Zongsoft/framework) framework.

It provides the client-side upgrade workflow, the out-of-process deployment executable, the server-side release repository, and the command-line tool used to create, validate, and publish release packages.

## Components

Component | Package or project | Responsibility
---------|--------------------|---------------
Client upgrader | [Zongsoft.Upgrading.Upgrader](upgrader/README.md) | Discovers newer releases, downloads packages, verifies checksums, extracts files, writes the deployment descriptor, starts the deployer, and exits the current application.
Client deployer | [Zongsoft.Upgrading.Deployer](deployer/README.md) | A Native AOT executable that waits for the host application to exit, copies the extracted files into the application directory, runs deployment executors, restarts the application, and terminates.
Web package manager | [Zongsoft.Upgrading.Web](web/README.md) | Hosts release metadata and package files, manages applications/releases/instances, and serves the discovery and download endpoints used by the upgrader.
Package tool | [Zongsoft.Tools.Upgrader](tool/README.md) | Creates upgrade ZIP packages and `.manifest` files, recalculates checksums, and publishes packages to supported channels.

## Upgrade Flow

1. Build or publish the target application.
2. Use `dotnet-upgrade pack` to create the release package and `.manifest` file.
3. Publish the package to a file storage location or to the Web package manager.
4. The running application loads `Zongsoft.Upgrading.Upgrader` and periodically checks the configured `File` or `Web` channel.
5. The upgrader selects releases matching the application name, edition, platform, architecture, and version range.
6. The selected package is downloaded, checksum-verified, and extracted to a temporary `.app` directory.
7. A `.deployment` deployment descriptor is written in the application directory.
8. The upgrader starts `Zongsoft.Upgrading.Deployer` and shuts down the current application.
9. The deployer waits for the host process to exit, optionally cleans the application directory for full releases, copies the extracted files, runs `Deploying` and `Deployed` executors, and restarts the application.

## Release Package

Each release contains:

- a ZIP package with application files;
- a `.manifest` metadata file generated by the package tool;
- optional tags, title, summary, description, custom properties, and deployment executors.

The release identity is composed from:

- application name;
- edition, such as `stable`, `community`, `standard`, `professional`, or `enterprise`;
- version;
- platform, such as `windows` or `linux`;
- architecture, such as `x64`, `x32`, `arm64`, or `arm32`;
- release kind, either `Fully` or `Delta`.

### Manifest Example

```xml
<?xml version="1.0" encoding="UTF-8"?>

<release
	name="Zongsoft.Hosting.Terminal"
	kind="Fully"
	edition=""
	version="1.1.0"
	size="123456"
	path="Zongsoft.Hosting.Terminal@1.1.0_win-x64.zip"
	checksum="SHA1:1234567890ABCDEF"
	platform="Windows"
	architecture="X64"
	deprecated="false"
	creation="2026-04-10T10:30:59">

	<title></title>
	<summary></summary>
	<description></description>

	<tags>
		<tag></tag>
	</tags>

	<properties>
		<property name="" type="">
		<![CDATA[
		]]>
		</property>
	</properties>

	<executors>
		<executor event="">
		<![CDATA[
		]]>
		</executor>
	</executors>
</release>
```

## Release Kinds

- `Fully`: a full release. During deployment the deployer cleans the application directory before copying the extracted package files.
- `Delta`: an incremental release. The deployer copies the extracted files over the existing application directory without first cleaning it.

## Channels

The client upgrader currently supports two discovery/download channels:

- `File`: reads manifest and package files from a configured file-system URL, including Zongsoft file-system providers such as `zfs.s3:/...`.
- `Web`: calls the Web package manager endpoint and downloads package files from URLs returned by the server.

The package tool currently publishes to:

- `amazon.s3`.
- `web`.

## Executor Commands

Release manifests can contain executor commands that run during deployment events. Both the upgrader and deployer register the built-in commands:

- `Copy`
- `Move`
- `Link`
- `Delete`

The deployer invokes executors on `Deploying` and `Deployed`, which makes them useful for service-file updates, symlink maintenance, cleanup, or other controlled deployment tasks.
